Deflate 알고리즘의 NPNCB 주입 취약점 분석

Analysis of Malicious Code Injection Vulnerability in the Deflate Algorithm

Author: Jung-hoon Kim (CEO of BinaryLab)

1. 취약점 핵심 요약

본 연구는 전 세계적으로 가장 널리 사용되는 압축 알고리즘인 Deflate(RFC 1951)에서 비 압축 블록의 구조적 특징을 이용한 데이터 은닉 및 악성코드 실행 기법을 다룹니다[cite: 10, 21]. 공격자는 정상적인 압축 데이터 블록 사이에 NPNCB(No-Payload Non-Compressed Block)를 삽입하여 보안 솔루션을 우회할 수 있습니다[cite: 10].

2. 기술적 원리 (Mechanism)

A. DBA (Disposed Bit Area)의 정의

Deflate 알고리즘은 비 압축 블록을 생성할 때, 헤더 이후의 데이터가 바이트 경계(Byte Boundary)에서 시작하도록 설계되었습니다[cite: 11, 37]. 이때 이전 블록의 끝 지점에 따라 0~7비트의 여유 공간이 발생하는데, 이를 본 연구에서 DBA라고 명명하였습니다[cite: 11, 39].

비 압축 블록 헤더 구조: [BFINAL(1bit)] + [BTYPE(2bit: 00)] + [DBA(0-7bit)] + [LEN(16bit)] + [NLEN(16bit)]

RFC 1951 표준에 따르면 DBA 영역의 값은 무시되므로(ignored), 공격자는 여기에 임의의 데이터를 채워 넣을 수 있습니다[cite: 38].

B. NPNCB (No-Payload Non-Compressed Block)

NPNCB는 실제 저장할 원본 데이터가 없어 LEN 값이 0인 특수한 비 압축 블록입니다[cite: 101, 125]. 이 블록은 압축 결과물에 어떠한 출력 데이터도 남기지 않으면서 오직 DBA 영역만을 확보하는 용도로 사용됩니다.

3. 공격 시나리오 (Attack Scenario)

1. 은닉 (Hiding): 악성 코드를 비트 단위로 쪼개어 수만 개의 NPNCB 내 DBA 영역에 분산 저장합니다[cite: 301, 328].
2. 우회 (Bypassing): NPNCB는 해제된 데이터가 없으므로 파일의 전체 CRC(Cyclic Redundancy Check) 값에 영향을 주지 않습니다. 따라서 일반 백신이나 압축 프로그램은 변조를 감지하지 못합니다[cite: 143, 266].
3. 실행 (Execution): 특수하게 설계된 악의적 디코더가 압축을 풀면서 DBA의 비트들을 다시 조합하여 메모리 상에서 악성 코드를 복원 및 실행합니다[cite: 290, 336].

4. 결론 및 시사점

이 취약점은 ZIP뿐만 아니라 APK, PDF, XLSB 등 Deflate를 사용하는 수많은 파일 포맷에 적용될 수 있습니다[cite: 294, 295]. 단순한 데이터 은닉을 넘어 스테가노그래피 기술로 활용될 수 있으며, 이를 차단하기 위한 새로운 보안 검증 알고리즘이 필요합니다[cite: 390, 395].

1. Research Summary

This study analyzes a vulnerability in the Deflate algorithm (RFC 1951), the world's most popular compression method[cite: 2, 21]. It demonstrates how NPNCB (No-Payload Non-Compressed Block) can be used to hide and execute malicious code between normal data blocks[cite: 10, 16].

2. Technical Mechanism

A. DBA (Disposed Bit Area)

When generating non-compressed blocks, Deflate ensures data starts at a byte boundary[cite: 37, 39]. This creates a gap of 0-7 bits, defined as DBA in this research[cite: 11, 47].

Non-compressed Block Header: [BFINAL(1bit)] + [BTYPE(2bit: 00)] + [DBA(0-7bit)] + [LEN(16bit)] + [NLEN(16bit)]

Standard decoders ignore data in the DBA area per RFC 1951 specifications[cite: 38].

B. NPNCB Characteristics

NPNCB is a special block where LEN = 0[cite: 10, 125]. It produces no output during decompression but serves as a carrier for hidden data in the DBA[cite: 14, 143].
