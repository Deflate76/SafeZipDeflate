Deflate 압축 알고리즘에서
악성코드 주입 취약점 분석

김정훈 (Jung-hoon Kim)
바이너리랩 주식회사 (BinaryLab CEO)
Journal of The Korea Institute of Information Security & Cryptology
VOL.32, NO.5, Oct. 2022 | ISSN 1598-3986

요약 (Abstract)

본 연구를 통해 매우 대중적인 압축 알고리즘인 Deflate 알고리즘을 통해 생성되는 3가지 유형의 압축 데이터 블록 가운데 원본 데이터 없는 비 압축 블록(No-Payload Non-Compressed Block; NPNCB) 유형을 임의로 생성하여 정상적인 압축 블록 사이에 미리 설계된 공격 시나리오에 따라 삽입하는 방법을 통해 악의적 코드 또는 임의의 데이터를 은닉하는 취약점을 발견하였습니다.

비 압축 블록의 헤더에는 바이트 정렬(Byte Align)을 위해서만 존재하는 데이터 영역이 존재하며, 본 연구에서는 이 영역을 DBA(Disposed Bit Area)라고 명명하였습니다. 이러한 DBA 영역에 다양한 악성 코드와 악의적 데이터를 숨길 수 있었으며, 기존 상용 프로그램에서 경고 없이 압축 해제되고 악의적 디코더로 해독하여 임의 코드를 실행할 수 있음을 입증했습니다.

1. Deflate 압축 블록과 DBA 개념

Deflate 알고리즘(RFC 1951)은 압축 대상의 특성에 따라 세 가지 블록(Dynamic, Fixed, Non-compressed)을 생성합니다. 이 중 '비 압축 블록(Non-compressed block)'은 압축을 포기하고 원본 데이터를 그대로 저장할 때 사용됩니다.

바이트 정렬과 여유 공간(DBA)

비 압축 블록은 바이트 경계에서 시작해야 하므로, 이전 압축 블록이 끝나는 비트 위치에 따라 0~7비트의 가변적인 여유 공간이 발생합니다.

// Non-compressed Block Header Structure
[BFINAL: 1bit] + [BTYPE: 2bits (00)] + [DBA: 0~7bits padding] + [LEN: 16bits] + [NLEN: 16bits]

RFC 1951 표준에 따르면 이 패딩 영역(DBA)의 데이터는 디코더에 의해 완전히 무시(ignored)됩니다. 상용 프로그램은 이 영역을 0으로 초기화하지만, 공격자는 이 공간에 악의적인 페이로드를 기록할 수 있습니다.

2. NPNCB (No-Payload Non-Compressed Block)

NPNCB는 LEN = 0인 특수한 형태의 비 압축 블록입니다. 원본 데이터를 전혀 포함하지 않기 때문에, 압축 해제 시 출력되는 데이터 스트림에 아무런 영향을 주지 않습니다.

  • CRC 우회 (CRC Evasion): 기존 보안/압축 소프트웨어는 '해제된 원본 데이터'를 기준으로 CRC 무결성을 검사합니다. NPNCB는 출력 데이터가 없으므로 파일이 변조되었음에도 CRC 오류를 발생시키지 않습니다.
  • 패킷/스트림 제어용 악용: TLS, SSH, PPP 등 연속적인 데이터 스트림을 패킷 단위로 분할할 때 사용되는 정상적인 강제 분할(Flush) 기법을 모방하여 의심을 피합니다.

3. 공격 사례 및 취약점 증명

본 연구는 7,680 바이트 크기의 Windows 실행 파일(악성 코드)을 분할하여 Zip 압축 파일 내부에 주입하는 모의 공격을 수행했습니다.

// 1 바이트의 악성코드를 5비트 / 3비트로 분할하여 2개의 NPNCB DBA에 은닉
Payload Size: 7,680 bytes
Required NPNCBs: 15,360 blocks
Total Injected Hidden Data: ~76,800 bytes

실험 결과:

  • 알집(ALZip), 반디집(Bandizip), WinRAR 등 대중적인 상용 디코더는 삽입된 수만 개의 NPNCB를 모두 무시하고 원본 엑셀/오디오 파일을 경고 없이 정상적으로 압축 해제했습니다.
  • 오염된 Zip 파일을 악의적인 디코더(설계된 공격 도구)로 읽어들일 경우, 숨겨진 DBA 비트들을 메모리에서 재조합하여 악성 코드를 즉각적으로 실행(Execute)하는 데 성공했습니다.

Malicious Code Injection Vulnerability Analysis
in the Deflate Algorithm

Jung-hoon Kim
BinaryLab (CEO)
Journal of The Korea Institute of Information Security & Cryptology
VOL.32, NO.5, Oct. 2022 | ISSN 1598-3986

Abstract

Through this study, we discovered that among the three types of compressed data blocks generated by the Deflate algorithm, a specific type called the No-Payload Non-Compressed Block (NPNCB) can be arbitrarily generated and inserted between normal compressed blocks to hide malicious code or arbitrary data.

In the header of a non-compressed block, there is a data area that exists solely for byte alignment. We named this area the DBA (Disposed Bit Area). We successfully demonstrated that an attacker can hide various malicious codes within the DBA. Even when contaminated NPNCB blocks are inserted, commercial programs decode the zip file normally without any warnings, and a malicious decoder can extract and execute the hidden code.

1. Deflate Compression Blocks & The DBA Concept

The Deflate algorithm (RFC 1951) generates three types of blocks: Dynamic, Fixed, and Non-compressed. Non-compressed blocks are utilized when compression is inefficient, storing literal data as-is.

Byte Alignment and the DBA

Because non-compressed blocks must start on a byte boundary, a variable padding of 0 to 7 bits is appended to the end of the previous block's data.

// Non-compressed Block Header Structure
[BFINAL: 1bit] + [BTYPE: 2bits (00)] + [DBA: 0~7bits padding] + [LEN: 16bits] + [NLEN: 16bits]

According to the RFC 1951 specification, decoders strictly ignore the data inside this padding (DBA). While standard encoders initialize this area with zeros, attackers can inject malicious payloads into this blind spot.

2. The NPNCB (No-Payload Non-Compressed Block)

An NPNCB is a specialized block where LEN = 0. Because it contains zero literal data, it outputs absolutely nothing during the decompression stream.

  • CRC Evasion: Traditional security and archiving tools verify file integrity by checking the CRC of the decompressed output. Since NPNCBs produce no output, the CRC remains completely unaffected, evading detection.
  • Exploiting Stream Flush Mechanics: NPNCBs are legitimately used in protocols like TLS, SSH, and PPP to force byte-aligned flushes. Injecting them mimics benign network behavior, making forensic analysis difficult.

3. Attack Scenario & Proof of Concept

In our experiment, a 7,680-byte Windows executable malware was fragmented and injected into a standard Zip archive.

// Splitting 1 byte of malware into 5-bit & 3-bit chunks across 2 NPNCBs
Payload Size: 7,680 bytes
Required NPNCBs: 15,360 blocks
Total Injected Hidden Data: ~76,800 bytes

Results:

  • Major commercial archivers (ALZip, Bandizip, WinRAR) completely ignored the thousands of injected NPNCBs, extracting the original benign files with zero integrity warnings.
  • When the contaminated Zip file was processed by a custom malicious decoder, it successfully reassembled the hidden DBA bits in memory and executed the payload immediately.
